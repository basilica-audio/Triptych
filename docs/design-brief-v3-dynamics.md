# Triptych — Design Brief v3 Addendum: Hybrid Dynamic Multiband (v0.3.0)

Addendum to `docs/design-brief.md` (v0.2.0, still binding for everything not superseded here). Scope: upgrade Triptych's per-band compressor from a **downward-only** compressor to a **true dynamic multiband processor** - the paradigm decided for this pass is **Hybrid à la Pro-MB**: keep the existing single Ratio parameter, extend it below 1:1 for upward compression/expansion, add a per-band Range (maximum gain-change clamp), and file downward expansion (gating) as an explicitly deferred, separate capability rather than bolting it onto a model that doesn't cleanly fit it. **No brand or person names in parameters, UI, or marketing copy** - research citations only in this document and `docs/research-notes.md`.

## Why v0.2.0 falls short of "dynamic multiband"

v0.2.0 gave every band a soft-knee gain computer (`src/dsp/KneeGainComputer.{h,cpp}`), but its Ratio parameter was hard-floored at 1:1 and its gain-reduction formula returned exactly 0 dB (an unconditional bypass) for *any* `ratio <= 1.0`. That means v0.2.0 can only ever make a band **quieter** above threshold - it cannot lift a signal, cannot restore dynamics to over-compressed material, and has no safety valve against the runaway gain that an aggressive setting can otherwise produce. Every reference-class unit surveyed for v0.2.0 (Weiss DS1-MK3, FabFilter Pro-MB) documents ratio ranges that extend *below* 1:1 specifically for this reason - v0.2.0's hard floor at 1:1 is a real, previously-undocumented gap versus that reference class, not a deliberate design choice; it was simply how `juce::dsp::Compressor`'s own formula (which v0.2.0's `KneeGainComputer` was built to reproduce bit-for-bit at Knee = 0%) happens to behave for `ratio <= 1`.

## The hybrid paradigm

```
in → LR4 crossovers (unchanged) → BandComp Low/Mid/High:
       Knee → Threshold/Ratio gain computer (now spans 0.2:1-20:1) → Range clamp → Attack/Release → Makeup
       → (High only) optional Limiter → Mute/Solo gate → Sum → Output trim
```

1. **Ratio extended below 1:1 (upward compression/expansion).** The parameter's range widens from **1:1-20:1** to **0.2:1-20:1** - the lower bound is the Weiss DS1-MK3 manual's own documented endpoint ("adjustable from 1000:1 to 1:5"; 1:5 = 0.2), not an arbitrary number (`docs/research-notes.md`). The *same* closed-form transfer curve v0.2.0 already used for downward compression (`thresholdDb + delta * ratioInverse` above threshold, or its quadratic soft-knee blend) is now evaluated continuously for ratio values on both sides of 1:1, rather than being replaced by a different formula or a naive inversion. For ratio > 1 this cuts signal above threshold (downward compression, unchanged from v0.2.0); for ratio < 1 the exact same formula *boosts* signal above threshold instead, because the `(ratioInverse - 1)` term that scales the transition flips sign. Material below threshold remains untouched in both regimes, exactly as in v0.2.0 - this is not a "raise the noise floor" style upward compressor (that would need a second, independent threshold; see the honesty section), it is a continuous, symmetric extension of the *existing* single-threshold curve, matching how Weiss's own continuous ratio parameter is documented to work.
2. **1:1 is an exact, bit-for-bit null point.** `ratio == 1.0` is special-cased at the top of both `computeStaticGainReductionDb()` and `computeGainLinear()` to return `0.0`/`1.0` unconditionally, independent of threshold, knee, or Range - not relying on floating-point coincidence in the general formula (which is *extremely* close but not guaranteed bit-exact at that boundary due to ordinary IEEE754 rounding). This is the taper the brief calls for: 1:1 is not just "close to a null", it is a hard, tested guarantee.
3. **Range (new): a per-band maximum gain-change clamp, in dB, applying to both directions.** `xRangeEnabled` (bool, default **off**) + `xRange` (float, **0-30 dB**, default **12 dB**, applied only while enabled). When engaged, the band's gain change - whether a downward cut or an upward boost - is clamped to at most `xRange` dB in magnitude. This is the reference-class safety valve (FabFilter Pro-MB's "Range knob limits the maximum amount of applied gain change", fetched 2026-07-18, `docs/research-notes.md`) that makes an aggressive Ratio setting - especially the new upward regime, whose gain change is otherwise unbounded as input rises further above threshold - usable rather than a runaway. Off by default: a fresh v0.3.0 band, or a band whose Range API is never touched at all, is unaffected.
4. **Downward expansion (gating) is deliberately deferred, not bolted on.** Pro-MB reaches a genuine per-band gate by crossing its Dynamics Mode (Compress/Expand) button with a *signed* Range knob - a structurally different control surface from Triptych's hybrid model, which uses one continuous Ratio and an unsigned, bidirectional Range. A clean downward expander needs its own, independent threshold (typically well below the compressor's own threshold - "duck anything quieter than X", not "everything above threshold, scaled by ratio") - retrofitting that onto the existing single-threshold Ratio/Range curve would either produce confusing, non-orthogonal parameter interactions or require a second threshold parameter per band, which is a larger, separate piece of design work. See the honesty section below; filed as a tracked GitHub issue rather than implemented partially.

## Parameter changes

| Parameter | v0.2.0 range | v0.3.0 range | Default | Change |
|---|---|---|---|---|
| `lowRatio` / `midRatio` / `highRatio` | 1.0 - 20.0 | **0.2 - 20.0** | 2.5 / 1.8 / 2.0 (unchanged) | Lower bound widened for upward compression/expansion. Skewed (`setSkewForCentre(1.0f)`) so 1:1 sits at the knob's centre travel. |
| `lowRangeEnabled` / `midRangeEnabled` / `highRangeEnabled` *(new)* | - | Off / On | **Off** | New. Off by default so v0.2.0 behaviour is reproduced exactly unless a user deliberately opts in. |
| `lowRange` / `midRange` / `highRange` *(new)* | - | 0 - 30 dB | **12 dB** | New. Only takes effect while the corresponding `RangeEnabled` is on. |

Every other v0.2.0 parameter (Threshold, Knee, Attack, Release, Makeup, crossover splits, Mute/Solo, High-band limiter, Output) is **unchanged** - IDs, ranges, and defaults all carry over verbatim, per `src/params/ParameterIds.h`'s frozen-ID contract.

## Defaults: bit-identical to v0.2.0

A fresh v0.3.0 instance is bit-identical to v0.2.0 at its shipped defaults:

- Every Ratio default (2.5 / 1.8 / 2.0) sits well inside the new, wider 0.2-20.0 range - the range widening is purely additive at the extremes, the defaults themselves don't move.
- `RangeEnabled` defaults to **off** on every band, which routes `KneeGainComputer` to its `unlimitedRangeDb` sentinel (500 dB - see below) - a value no realistic combination of this engine's own Threshold/Ratio ranges can reach, so the Range clamp is a genuine, verified no-op at its default state, not just "usually doesn't matter".
- A v0.2.0-shaped saved session or preset (missing all six new parameter IDs) loads tolerantly via `AudioProcessorValueTreeState::replaceState()`'s existing "missing ID keeps current (= default) value" behaviour - `RangeEnabled` resolves to off and `Range` to 12 dB, exactly as a fresh instance would, with no special-case migration code needed (the same mechanism the v0.2.0 Knee migration already relied on).

## Numerical safety: the `unlimitedRangeDb` sentinel

`KneeGainComputer::unlimitedRangeDb` (500 dB) is a defensive ceiling applied **unconditionally**, regardless of the caller's own Range setting - `computeStaticGainReductionDb()`'s result is always clamped to `+-min(rangeDb, unlimitedRangeDb)`. This exists purely to keep `juce::Decibels::decibelsToGain()` inside `float`'s representable range at pathological (far outside any realistic operating point) parameter/envelope combinations; 500 dB is comfortably below the ~770 dB point where `decibelsToGain()` would overflow a 32-bit float, and comfortably above the largest gain change this engine's own actual parameter ranges can produce even at their extremes applied to a generously-hot +24 dBFS input (~420 dB - see the constant's own doc comment in `src/dsp/KneeGainComputer.h`).

This ceiling bounds the gain change **per processing pass**, applied to genuinely fresh program material each call, as every real DAW host does. It does not (and is not intended to) prevent divergence under a *literal, sustained audio feedback loop* - repeatedly feeding an upward-ratio band's own output back into itself as if it were new input is a positive-feedback scenario with a repeatedly-reapplied bounded-but-nonzero boost each pass, structurally identical to a delay effect with feedback over 100% or a self-oscillating resonant filter; it is not something a single-pass, per-block static transfer curve is responsible for damping, and it is outside the "handle any single call safely" real-time-safety contract this engine is otherwise held to. See `tests/RobustnessTests.cpp`'s "Extreme upward ratio" test for the explicit reasoning and methodology (fresh audio supplied every block, matching real host behaviour).

## Test guarantees (Catch2; all v0.2.0 cases kept, extended with the following)

1. **1:1 exact null:** `ratio == 1.0` produces exactly `0.0` dB gain reduction / exactly `1.0` linear gain, independent of Threshold, Knee, and Range, across a wide sweep (`KneeGainComputerTests.cpp`).
2. **Upward transfer curve, quantitative:** for `ratio < 1.0`, hard knee, a signal sitting `inputDb` above threshold is boosted by exactly `(inputDb - thresholdDb) * (ratioInverse - 1.0)` dB - the same closed-form the downward case already used, evaluated on the other side of 1:1 (`KneeGainComputerTests.cpp`).
3. **Range clamp:** gain change never exceeds `+-Range` dB in either direction, across a sweep spanning both the downward and upward regimes, at both the pure-math (`KneeGainComputerTests.cpp`) and envelope-integrated, real-audio (`BandCompressorTests.cpp`) levels.
4. **Range default is a true no-op:** the 4-argument `computeGainLinear()`/`computeStaticGainReductionDb()` overloads (used by every pre-v0.3.0 test) are bit-exact-equal to the 6-argument form called with the explicit `unlimitedRangeDb` sentinel; a `BandCompressor` whose Range API is never touched at all produces byte-identical output to one that explicitly disables it.
5. **v0.2.0-state import null:** a v0.2.0-shaped saved session (missing all six new parameter IDs) loads without crash/assertion, with `RangeEnabled` resolving to off and `Range` to its 12 dB default on every band (`StateTests.cpp`).
6. **Bit-identical fresh-instance defaults:** a fresh v0.3.0 processor's v0.2.0-era parameter defaults are pinned against the literal v0.2.0 numbers; every new parameter defaults to a neutral/off state; and a `BandCompressor` driven through the v0.3.0 API surface at shipped defaults (explicitly calling `setRangeEnabled(false)`) produces the same measured gain reduction, on real audio, as one that never calls the Range API at all (`StateTests.cpp`).
7. **NaN/Inf robustness:** the full parameter sweep is extended to cover ratios down to 0.2, Range enabled/disabled at both its 0 dB and 30 dB extremes, and the specific "extreme upward ratio + deepest threshold + Range disabled" combination the Range feature exists to make usable (`KneeGainComputerTests.cpp`, `BandCompressorTests.cpp`, `RobustnessTests.cpp`).
8. **Preset round-trip:** all eight factory presets (two extended to showcase upward compression/Range, per below) load, round-trip through save/load, and stay within their declared parameter ranges - extending the existing v0.2.0 preset test rather than replacing it (`PresetManagerTests.cpp`).

## Factory presets updated to showcase the hybrid paradigm

Per the brief, 1-2 presets were extended (not replaced) to demonstrate the new capability - the existing eight all remain valid, loadable presets with their original philosophical intent intact:

- **Density Glue** (Master): Mid band's Ratio changed from 1.2:1 (gentle downward) to **0.7:1 (genuine upward)** - the "knit together" density philosophy this preset already represented is now achieved by *lifting* the Mid band's presence-region content toward cohesion rather than only cutting its peaks. Range engaged on all three bands at a modest **8 dB**, since an upward Mid band combined with this preset's already-low threshold benefits from the safety margin the moment a user pushes further from the preset's own starting point.
- **Parallel-Style Density** (Master): all three bands' Ratio pushed to **0.6:1 (upward)**, deepening the density effect the preset's name already promised, now via genuine upward compression rather than a very-low-ratio downward compressor. Range engaged on all three bands at **10 dB**, explicitly because this preset's deep (-38 dB) threshold combined with a strongly upward ratio would otherwise be able to produce a much larger boost than is musically useful on full-scale peaks - Range is what keeps this preset's extremity usable rather than a coincidence of the specific test signal it was voiced against.

The remaining six presets (Default, Peak Control, Low-End Tighten, De-Harsh Highs, Mastering Safety Ceiling, Hard Limiter Ceiling) are otherwise unchanged; their JSON files gain the six new parameter keys at their neutral v0.3.0 defaults (`RangeEnabled: false`, `Range: 12.0`) purely for the "complete snapshot" convention the preset format already follows - loading them produces identical sound to v0.2.0.

## Honesty & framing

- This voicing is **research-derived** (Weiss DS1-MK3's documented continuous ratio range; FabFilter Pro-MB's Range concept, fetched live for this pass), **not measured against reference hardware** - the same framing v0.2.0 already used, extended here rather than superseded.
- **Not a Pro-MB clone.** Pro-MB reaches four distinct behaviours (downward/upward compression, downward/upward expansion) via a Dynamics Mode button crossed with a signed Range knob. Triptych's v0.3.0 hybrid is a deliberately different, simpler design: one continuous Ratio spanning both regimes (Weiss's convention) plus an unsigned, bidirectional Range clamp (Pro-MB's Range *concept*, not its exact control surface). This is a considered design choice, not an incomplete implementation of Pro-MB's model - see `docs/research-notes.md`'s v0.3.0 addendum for the explicit reasoning.
- **No sourced numeric Range default.** Neither Pro-MB's help documentation nor the previously-captured Waves C6 "Range" framing (search-synthesis only) documents a specific dB range or default for a maximum-gain-change control. The 0-30 dB range and 12 dB default are a **reasoned engineering choice**, explicitly not claimed as sourced.
- **Explicitly out of scope for v0.3.0**, tracked as GitHub issues rather than implied as parity with the reference class:
  - **Downward expansion / gating.** A clean per-band gate needs a second, independent threshold that the current single-threshold Ratio/Range model doesn't cleanly accommodate - see "The hybrid paradigm" above. Filed as a new issue rather than bolted onto this pass.
  - **External sidechain** and **adjustable crossover slopes** - already tracked under issue #1 since v0.1.0 for the documented structural reasons (`docs/architecture.md`'s "Deferred from M1" section); unchanged by this pass, noted with a comment on that issue rather than a duplicate.
  - **Per-band Mid/Side processing** - identified as a gap in v0.2.0's own research pass but not previously given its own tracking issue; filed as a new issue in this pass alongside the downward-expansion one, since v0.3.0's research touched the same reference-class units (Weiss DS1-MK3 ships M/S as a headline feature) without addressing it.
  - **RMS/Peak-selectable detector mode** and **program-dependent/auto-release** - unchanged from v0.2.0's own deferral (`docs/architecture.md`'s "Deferred from v0.2.0" section); this pass's scope was the Ratio/Range dynamics model, not the envelope detector.

## Versioning

Ships as **v0.3.0**. Breaking parameter changes are acceptable pre-1.0 (Ratio's `NormalisableRange` lower bound changes from 1.0 to 0.2, and six new parameter IDs are added); existing parameter IDs, their defaults, and every other range are otherwise preserved - additive + one range-widening, not a rename or a default change. State migration = **tolerant import**, verified explicitly (see test guarantee #5). CHANGELOG documents the Ratio range extension and the new Range parameter prominently, with a pointer to this document and `docs/research-notes.md`'s v0.3.0 addendum for the sourcing.
